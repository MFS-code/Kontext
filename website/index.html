kontext

Agents are workloads.

Kontext is a Kubernetes control plane for AI agents. An Agent is a custom resource reconciled into real Pods: apply it, watch runtime logs, read the result. No new platform to learn. 

# install a published release (no clone, no image build)
VERSION=v0.1.0-alpha.1
$ kubectl apply -f \
    "https://github.com/MFS-code/Kontext/releases/download/${VERSION}/install.yaml"
$ kubectl rollout status deployment/controller-manager \
    -n kontext-system --timeout=120s

# run.yaml — keyless echo runtime
apiVersion: kontext.dev/v1alpha1
kind: AgentRun
metadata:
  name: review
spec:
  goal: "Summarize the failure modes in this release"
  provider: echo
  model: echo-model
  runtime:
    image: ghcr.io/mfs-code/kontext-echo:v0.1.0-alpha.1
  budget:
    wallclock: 5m

$ kubectl apply -f run.yaml
$ kubectl logs -f run-review                 # runtime logs, live
$ kubectl get agentrun review \
    -o jsonpath='{.status.result}'           # the answer

Why Kubernetes

Scheduling, secrets, RBAC, budgets, logs, restarts: agents need what Kubernetes already does. Kontext adds two resources, Agent and AgentRun, and leaves the rest to the cluster. Service agents are re-cast automatically when they die. Wallclock budgets are enforced by the controller. Results live in status, queryable like any other object.

Bring your own runtime

Any container image that reads a few KONTEXT_* env vars and optionally writes a result envelope is a valid agent. Kontext never inspects what the agent does beyond that I/O boundary. Releases ship a keyless echo runtime for control-plane checks and a maintained Go reference runtime for model-backed work.

Budgets that mean something

budget.wallclock is enforced by the controller: after expiry the run becomes BudgetExceeded and the Pod is deleted. Token and dollar fields are reported by the runtime when available; they are measurements, not hard stops, unless your runtime enforces them.

Alpha boundaries

v1alpha1 is unstable on purpose. Matching API version labels are not compatibility guarantees between Kontext releases. Service mode and standalone AgentRuns are implemented; Task and Scheduled modes exist in the schema but are not reconciled yet. Read the release notes before upgrading.
